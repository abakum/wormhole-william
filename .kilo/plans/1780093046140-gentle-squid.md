# План миграции на новый API туннелей

## Обзор изменений

С последнего коммита (`9bf1852`) рабочий каталог содержит 4 изменённых файла. Главное изменение — **разделение `CreateTunnel` на две фазы** через новый метод `PrepareTunnel`. Это позволяет получить код wormhole **до** установки соединения, что критично для UX (код можно показать пользователю, пока peer ещё подключается).

---

## 1. Изменение публичного API

### 1.1 Новый метод `Client.PrepareTunnel`

**Было (старый `CreateTunnel`):**
```go
generatedCode, t, err := c.CreateTunnel(ctx, code)
// t — уже подключённый *tunnel.Tunnel
defer t.Close()
```

**Стало:**
```go
generatedCode, connect, err := c.PrepareTunnel(ctx, code)
// connect — функция, вызов которой блокируется до установки соединения
log.Printf("wormhole code: %s", generatedCode)  // показать код пользователю

t, err := connect()  // блокирует до подключения peer
if err != nil { ... }
defer t.Close()
```

**Сигнатура:**
```go
// Старая (сохранена как обёртка для обратной совместимости)
func (c *Client) CreateTunnel(ctx context.Context, code string) (string, *tunnel.Tunnel, error)

// Новая — двухфазная
func (c *Client) PrepareTunnel(ctx context.Context, code string) (string, func() (*tunnel.Tunnel, error), error)
```

### 1.2 `JoinTunnel` — без изменений сигнатуры

```go
_, t, err := c.JoinTunnel(ctx, code)  // работает как раньше
```

### 1.3 `tunnel.Tunnel` — без изменений

Методы `Serve`, `Forward`, `Dial`, `Listen`, `Close` не изменились.

## 2. Что поменялось внутри (не влияет на публичный API, но важно знать)

| Область | Изменение |
|---------|-----------|
| `negotiateRelay` | Теперь возвращает `(relayAddr, label string)` вместо `string` |
| `countTransitHints` | Закомментирован (не удалялся) |
| Логирование | Большинство `log.Printf` внутри сессии/туннеля закомментированы для уменьшения шума; критичные сообщения оставлены |
| `CreateTunnel` | Реализован как обёртка над `PrepareTunnel` + немедленный `connect()` — полная обратная совместимость |
| `claimOrAllocateNameplate` | Перенесён из тела цикла `CreateTunnel` в `PrepareTunnel` (один вызов до возврата кода) |

## 3. Шаги миграции для потребителей API

### Вариант A: Минимальная миграция (обратная совместимость)

Ничего менять не нужно — старый `CreateTunnel` сохранён как обёртка. Код, использующий `CreateTunnel`, продолжит работать без изменений.

### Вариант B: Миграция на `PrepareTunnel` (рекомендуется)

**До:**
```go
generatedCode, t, err := c.CreateTunnel(ctx, code)
if err != nil {
    log.Fatalf("create tunnel: %v", err)
}
log.Printf("wormhole code: %s", generatedCode)
defer t.Close()
t.Serve(ctx, localAddr)
```

**После:**
```go
generatedCode, connect, err := c.PrepareTunnel(ctx, code)
if err != nil {
    log.Fatalf("prepare tunnel: %v", err)
}
log.Printf("wormhole code: %s", generatedCode)

t, err := connect()
if err != nil {
    log.Fatalf("connect tunnel: %v", err)
}
defer t.Close()
t.Serve(ctx, localAddr)
```

**Ключевое отличие:** между `PrepareTunnel` и `connect()` можно показать код пользователю (вывести в UI, отправить по SMS и т.д.), не дожидаясь установки транспортного соединения.

### Изменения в примерах

- `examples/wormhole-william-tunnel-dial/main.go` — уже мигрирован на `PrepareTunnel`
- `examples/wormhole-william-tunnel-bind/main.go` — использует `JoinTunnel`, изменений не требует

## 4. Чеклист миграции

1. **[ ] Заменить вызовы `CreateTunnel` → `PrepareTunnel`** в пользовательском коде, где нужно показать код до подключения
2. **[ ] Обработать ошибку от `connect()`** — это новая точка отказа после отображения кода
3. **[ ] Решить, что делать с показанным кодом** если `connect()` вернёт ошибку (код уже мог быть отправлен peer)
4. **[ ] Проверить таймауты** — если между `PrepareTunnel` и `connect()` есть задержка (ожидание ввода пользователя), убедиться что `ctx` позволяет peer'у подключиться
5. **[ ] Если используется `JoinTunnel`** — изменений не требуется

---

## 5. Риски и ограничения

- `CreateTunnel` теперь делает nameplate allocation в `PrepareTunnel`, а retry-loop — внутри `connect()`. Если `connect()` не вызывается, nameplate зависает в rendezvous до таймаута mailbox.
- `negotiateRelay` возвращает второй аргумент (label), который в текущем коде игнорируется (`_`). В будущем может использоваться для диагностики.
- Закомментированные `log.Printf` и `countTransitHints` — технический долг. Рекомендуется либо удалить, либо заменить на `slog.Debug`.
